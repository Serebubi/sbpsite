# AGENT.md

## Что это за проект

`sbpsite-main` — монорепозиторий, в котором сейчас живут две связанные, но пока осознанно разделенные части:

1. `Сарма Экспресс` — новая публичная промо/landing-поверхность.
2. `SUPERBOX` — отдельное transactional-приложение с flow заказов, отслеживания, тарифов, поддержки и отправки по РФ.

Ключевой принцип текущего этапа: **не сливать `Сарма Экспресс` и `SUPERBOX` в одну визуальную систему без явной просьбы пользователя**. Сейчас связь между ними строится через точечные переходы по URL.

## Как общаться и в каком стиле работать

- Язык общения с пользователем: `русский`.
- Тон: прямой, краткий, без лишней теории.
- Подход: сначала проверить фактическое состояние файлов и роутов, потом менять код.
- Если задача визуальная: не «улучшать всё сразу», а вносить точные локальные правки под конкретный запрос.

## Текущая структура интерфейсов

### `Сарма Экспресс`

Это текущая главная сайта и отдельные маркетинговые страницы в том же стиле.

Основные страницы:

- `/` — главная `Сарма Экспресс`
- `/sarma-express` — та же главная
- `/calculator` — отдельная страница калькулятора
- `/pickup-points` — отдельная страница пунктов выдачи с картой

### `SUPERBOX`

Основное приложение находится на:

- `/superbox`

Поддерживаемые flow внутри `SUPERBOX`:

- `pickup_standard`
- `pickup_paid`
- `order_lookup`
- `home_delivery`
- `ship_russia`
- `cancel_order`
- `support`
- `tariffs`

Если нужно открыть конкретный экран `SUPERBOX`, это делается через:

- `/superbox?flow=<flow_id>`

## Актуальная логика верхнего меню `Сарма Экспресс`

Сейчас верхняя навигация `Сарма Экспресс` уже привязана к рабочим сценариям:

- `Калькулятор` → `/calculator`
- `Отслеживание` → `/superbox?flow=order_lookup`
- `Бизнесу` → `/superbox?flow=tariffs`
- `Доставка из интернет-магазинов` → `/superbox?flow=pickup_paid`
- `Отправления в РФ` → `/superbox?flow=ship_russia`
- `Пункты выдачи` → `/pickup-points`

Это важно: последние итерации пользователя были именно про привязку пунктов верхнего меню к нужным сценариям. При следующих задачах сначала проверять, не должен ли новый пункт вести в уже существующий `SUPERBOX flow`, а не в новую страницу.

## Что уже важно знать про реализацию

### 1. Общий sticky-хедер для `Сарма Экспресс` уже вынесен

Компонент:

- `frontend/components/sarma-express-header.tsx`

Он используется на:

- главной `Сарма Экспресс`
- странице калькулятора
- странице пунктов выдачи

Если пользователь просит:

- изменить высоту верхней плашки,
- поменять поведение при скролле,
- поправить навигационные кнопки,
- поменять размер логотипа в шапке,

то почти всегда правки нужно начинать именно с `frontend/components/sarma-express-header.tsx`.

### 2. Калькулятор пока визуальный, без реальных расчетов

Компонент:

- `frontend/components/delivery-calculator-page.tsx`

Сейчас это только подготовленный интерфейс. Формулы расчета стоимости пользователь еще не дал. Нельзя придумывать тарифную логику самостоятельно без отдельного задания.

### 3. Пункты выдачи уже собраны в отдельную страницу

Компонент:

- `frontend/components/pickup-points-page.tsx`

Там уже есть:

- список адресов слева,
- карта Яндекс Карт справа,
- переключение активной точки по клику,
- хардкод координат для точек, чтобы карта открывалась стабильно.

### 4. Яндекс Карты уже интегрированы, но есть нюанс

Для карты используется:

- `NEXT_PUBLIC_YANDEX_MAPS_API_KEY` в `frontend/.env.local`

Также:

- env-файлы добавлены в `.gitignore`
- в корне лежит `yandexmap.txt` с ключом, который пользователь когда-то положил вручную

Важно:

- локально карта отображается и точки видны
- при этом в консоли Яндекса есть предупреждение `Invalid API key`

То есть при проблемах с картой в будущем сначала проверять:

1. корректен ли ключ,
2. разрешены ли нужные домены (`localhost`, боевой домен),
3. не сломалась ли внешняя загрузка JS API.

### 5. `SUPERBOX` уже умеет открываться в нужный flow через query string

Логика чтения query-параметра `flow` уже реализована в:

- `frontend/components/superbox-app.tsx`

Поэтому если пользователь просит, чтобы кнопка вела на конкретный раздел `SUPERBOX`, не нужно делать новую архитектуру. Обычно достаточно перевести ссылку на:

- `/superbox?flow=<нужный_flow>`

## Важные ограничения

- Не ломать `SUPERBOX`, если задача касается только `Сарма Экспресс`.
- Не менять backend/Bitrix-логику без явной необходимости.
- Не удалять пользовательские исходники ассетов в корне проекта без просьбы.
- Не откатывать чужие незакоммиченные изменения.
- Не считать старые backend/shared изменения «мусором»: там есть актуальная работа по заказам и `pickupPoint`.

## Что делать первым в новом диалоге

1. Прочитать `AGENT.md`, `CURRENT_STATUS.md`, `PROJECT_CONTEXT.md`, `HANDOFF.md`.
2. Проверить `git status --short`.
3. Определить, относится задача к:
   - `Сарма Экспресс`,
   - `SUPERBOX`,
   - или к их связке через навигацию.
4. Если задача по шапке/переходам — начать с `frontend/components/sarma-express-header.tsx`.
5. Если задача про конкретный экран `SUPERBOX` — проверить существующий `flow` в `frontend/components/superbox-app.tsx`.

## Чего ожидает пользователь

Пользователь работает итеративно и часто дает короткие, точечные визуальные и навигационные задачи:

- поменять кнопку,
- привязать пункт меню,
- сделать отдельную страницу,
- подогнать шапку под скрин,
- добавить карту,
- открыть конкретный `SUPERBOX flow`.

В таких задачах правильный подход — маленькие точные правки, а не широкий рефакторинг.
