## Текущая продуктовая логика

### `Сарма Экспресс`

Используется как:

- главная страница,
- набор отдельных витринных страниц в том же стиле,
- входная точка в нужные сценарии `SUPERBOX`

### `SUPERBOX`

Остается отдельным приложением и открывает нужный экран по `flow`.

Актуально использовать:

- `pickup_paid` — самостоятельный заказ
- `pickup_standard` — заказ по ссылке / стандартный сценарий
- `order_lookup` — отслеживание
- `ship_russia` — отправка по РФ
- `tariffs` — тарифы

## На что обратить внимание следующему агенту

### 1. Рабочее дерево грязное

В проекте есть незакоммиченные изменения не только по новым страницам, но и по backend/shared. Перед новыми правками обязательно смотреть:

- `git status --short`

И ничего не откатывать без запроса пользователя.

### 2. Есть debug-артефакты и служебные локальные файлы

В корне сейчас есть, например:

- `yandexmap.txt`
- `output-pickup-points.png`

Они не являются ядром продукта. Удалять их без явной просьбы пользователя не нужно.

### 3. Если пользователь просит правку верхней плашки

Идти в:

- `frontend/components/sarma-express-header.tsx`

### 4. Если пользователь просит изменить поведение кнопки в верхнем меню

Сначала смотреть:

- `frontend/components/sarma-express-header.tsx`

Только если нужен новый `SUPERBOX`-экран, тогда уже проверять:

- `frontend/components/superbox-app.tsx`

## Что делать первым в новом диалоге

1. Прочитать `AGENT.md`, `PROJECT_CONTEXT.md`, `HANDOFF.md`.
2. Проверить `git status --short`.
3. Определить, задача про:
   - новую публичную часть,
   - `SUPERBOX`,
   - или про переход между ними.
4. Если задача про навигацию — начать с `frontend/components/sarma-express-header.tsx`.
5. Если задача про новый переход в `SUPERBOX` — сначала найти подходящий `flow`, а не делать новый экран с нуля.
